# Kube KTS CLI

Kube KTS は `kube-kts` コマンドで実行するコマンドラインツールです。Helm の型安全なラッパーであり、
Go テンプレートの代わりに Kotlin スクリプト (KTS) で Kubernetes リソースを記述し、標準的な Helm
Chart にレンダリングしたうえで、実際のクラスター操作を Helm に委譲します。

組み込みの `--help` 出力で足りない場合、本章が最初の参照先です。ヘルプ文は画面に収まるよう意図的に
短くしています。本章ではさらに踏み込み、各オプションの挙動・既定値・単位・相互作用を詳しく説明します。
各コマンドには専用ページがあり、その**すべて**のオプションを記載します。

## Helm との関係

Kube KTS は Helm を置き換えません。Helm への入力を準備します。典型的なコマンドは 2 つのフェーズで
動作します。

1. **レンダリングフェーズ（Kube KTS）:** リポジトリをスキャンし、Kotlin スクリプトをコンパイル・実行
   して、結果を標準的な Helm Chart としてターゲットディレクトリに書き出します。
2. **Helm フェーズ（委譲）:** Helm ベースのコマンド（`lint`、`template`、`install`、`upgrade`、`uninstall`）
   では、Kube KTS がそのレンダリング済み Chart に対して実際の `helm` を呼び出し、すべての Helm 専用
   オプションをそのまま転送します。

このため、これらのコマンドは Kube KTS オプション（レンダリングに影響）と Helm オプション（転送される）
の両方を受け付けます。最終的に Helm に渡るオプションには印が付きます。下記の「オプションマーカーの凡例」を参照してください。

!!! note "Helm のインストールが必要"
    `lint`、`template`、`install`、`upgrade`、`uninstall` には `helm` 実行ファイルが `PATH` 上にある必要が
    あります。`validate`、`compile`、`render` は Helm なしで動作します。

## 用法

```bash
kube-kts [グローバルオプション] <コマンド> <REPOSITORY> [TARGET] [コマンドオプション]
```

### リポジトリとターゲットのパス

- **`REPOSITORY`** は最初の位置引数で、Kube KTS リポジトリ（`*.spec.kts`、`*.lib.kts`、
  `values.yaml` などを含むディレクトリ）を指します。省略するとカレントディレクトリが使われます。
  パスが存在しない場合、Helm を呼ぶ前に即座に失敗します。
- **`TARGET`** はレンダリング系コマンド（`render`、`lint`、`template`、`install`、`upgrade`、`uninstall`）が
  使う任意の 2 番目の位置引数で、Chart のレンダリング先ディレクトリです。省略すると一時ディレクトリが
  作成され OS により後始末されます。生成された Chart を確認・再利用したい場合は明示的に指定してください。

### 終了コード

CLI は成功時に `0`、失敗時に 0 以外の終了コードを返します（無効なリポジトリ、レンダリング失敗、Helm
からの 0 以外の終了コードなど）。失敗を診断する際は `--exception` で完全なスタックトレースを確認できます。

## コマンド

| コマンド | ページ | 目的 |
|---|---|---|
| `validate` | [validate](validate.md) | KTS リポジトリを出力なしで検証する。 |
| `compile` | [compile](compile.md) | KTS リポジトリをオブジェクトインスタンスにコンパイルする。 |
| `render` | [render](render.md) | リポジトリをディスク上の標準 Helm Chart にレンダリングする。 |
| `lint` | [lint](lint.md) | レンダリング後に `helm lint` で lint する。 |
| `template` | [template](template.md) | レンダリングして `helm template` でマニフェストを出力する。 |
| `install` | [install](install.md) | レンダリングして `helm install` でクラスターにインストールする。 |
| `upgrade` | [upgrade](upgrade.md) | レンダリングして `helm upgrade` でリリースをアップグレード（またはインストール）する。 |
| `uninstall` | [uninstall](uninstall.md) | レンダリングして `helm uninstall` で 1 つ以上のリリースを削除する。 |

## これらのコマンドはレンダリングが必要か？（KTS の関連性）

コマンドが KTS リポジトリをレンダリングするかどうかで、Kotlin スクリプトがそのコマンドに関係するか —
つまりそのコマンドに **リポジトリ**（KTS・素の YAML・混在）を渡す必要があるか — が分かります。

- **レンダリングが必要 → リポジトリ必須（KTS 関連）:** `validate`、`compile`、`render`、`lint`、
  `template`、`install`、`upgrade` はいずれも *スキャン → コンパイル → レンダリング* を実行するため、
  KTS スクリプトに依存し、リポジトリを渡す必要があります。
- **レンダリングと無関係 → リポジトリ不要（KTS 非関連）:** 既存リリース・クラスター・リポジトリに対する
  操作（将来の `status`、`list`、`rollback` など）はレンダリング済み Chart を必要としないため、KTS
  スクリプトは関係しません。

!!! note "`uninstall` は特殊ケース"
    `uninstall` は名前だけでリリースを削除し、技術的にはレンダリング済み Chart を必要としません。一貫性の
    ため現在は Helm 呼び出し前にレンダリングしますが、機能的には KTS スクリプトは関係ありません。

## オプションマーカーの凡例

`--help` 出力では専用の列が各オプションの性質を示します。コマンドページのオプション表でも同じマーカーを
使い、ドキュメントがヘルプ画面と完全に一致するようにしています。

| マーカー | 意味 |
|---|---|
| `---->` | 基盤となる Helm CLI に転送されます。効果は Helm のものと同一です。 |
| `*` | 実験的で、将来のバージョンで変更・削除される可能性があります。`--experimental` が必要です。 |
| `!!!` | 危険 / セキュリティに関わります。使用前に説明をよく読んでください。 |

マーカーのないオプションは Kube KTS の内部でのみ使用され、Helm には届きません。

## グローバルオプション

これらはすべてのコマンドで利用でき、通常はコマンド名の**前**に置きます。

| オプション | マーカー | 説明 |
|---|---|---|
| `--debug` | `---->` | どのステップが実行され Helm にどの引数が渡されるかを含むデバッグ情報を出力します。Helm にも `--debug` として転送され、Helm 自体も詳細になります。想定外の挙動の最初の調査手段に最適です。 |
| `--verbose` | | trace レベルの詳細を含むすべての情報を出力します（`--debug` より多い）。スキャン/コンパイル/レンダリングの流れを追うのに便利です。Helm には転送されません。 |
| `--show-log-level` | | 各ログ行の前にレベル（INFO/DEBUG/…）を付けます。`--debug` または `--verbose` 有効時は自動的に有効になります。 |
| `--exception` | | エラー時に短いメッセージではなく完全な例外スタックトレースを出力します。バグ報告や分かりにくい失敗の診断に使います。 |
| `--experimental` | | 下記の実験的オプションを解放します。これがないと、実験的オプションを渡すとコマンドは説明付きで失敗します。 |
| `--unsafe` | `!!!` | Kotlin スクリプト内で `import` 文と完全修飾クラス名を許可します。スクリプトが任意の JVM コードを実行できるため、完全に信頼できるリポジトリでのみ有効にしてください。 |

## 実験的機能

実験的機能は安定しておらず、将来のバージョンで変更される可能性があります。`--experimental` フラグが必要で、
ないと何もする前にコマンドが失敗します。

| オプション | マーカー | 説明 |
|---|---|---|
| `--yaml-merge=TYPE` | `*` | YAML ドキュメント（例: ベースファイルとオーバーレイ）のマージに使うアルゴリズムを選びます。`HELM`（既定）は Helm 自身のマージ意味論を再現し結果が Helm と一致します。`INTERNAL` は Kube KTS 独自のアルゴリズムで、さらに `--yaml-array-merge` に従います。 |
| `--yaml-array-merge=TYPE` | `*` | 配列のマージ方法を制御し、`--yaml-merge=INTERNAL` の場合のみ有効です。`None` はベース配列をそのまま、`Replace`（既定）はオーバーレイ配列で置換、`AddFirst` はオーバーレイ項目を先頭に、`AddLast` は末尾に追加します。素の Helm では配列は常に置換されるため、その挙動を変えるためのつまみです。 |

## 値 (Values)

すべてのレンダリング系コマンド（`render`、`lint`、`template`、`install`、`upgrade`、`uninstall`）は値ファイルを
受け付けます。値は 2 つの段階で使われます。レンダリング中に Kotlin スクリプトから利用でき、**かつ**
Helm に渡されます。

| オプション | マーカー | 説明 |
|---|---|---|
| `-f`, `--values=FILE` | `---->` | YAML 値ファイル。Kotlin スクリプトの実行に使われ、さらに `-f` として Helm に転送されます。繰り返し可能で、指定順に重ねられ、後のファイルが前を上書きします。インラインの `--set*` 上書き（対応コマンド）はすべての `-f` ファイルより優先されます。ファイルが存在しないとコマンドは失敗します。 |
