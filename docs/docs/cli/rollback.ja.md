# rollback

```bash
kube-kts rollback <RELEASE> [REVISION] [オプション]
```

`helm rollback` を実行してリリースを以前のリビジョンへロールバックします。既存のリリースに対して動作するため、**リポジトリもレンダリング手順も不要**です——呼び出しは直接 Helm に転送されます。

## 動作の仕組み

1. リポジトリのスキャン・コンパイル・レンダリングは行われません。KTS スクリプトはここでは無関係です。
2. `helm rollback <RELEASE> [REVISION]` が、転送されるすべてのオプションを付加して実行されます。
3. Helm がロールバックを実行します。コマンドの終了コードは Helm の結果を反映します。

## パラメータ

| パラメータ | 必須 | 説明 |
|---|---|---|
| `RELEASE` | はい | ロールバックするリリース名。位置引数 `RELEASE` として Helm に転送されます。 |
| `REVISION` | いいえ | ロールバック先のリビジョン。省略時は直前のリビジョンへロールバックします。 |

## rollback オプション

| オプション | マーカー | 説明 |
|---|---|---|
| `--cleanup-on-fail` | `---->` | ロールバック失敗時、本ロールバックで作成された新規リソースの削除を許可します。 |
| `--dry-run` | `---->` | 変更を加えずロールバックをシミュレートします。 |
| `--force` | `---->` | 必要に応じて削除/再作成によりリソースを強制更新します。 |
| `--history-max=INT` | `---->` | リリースごとに保存する最大リビジョン数（`0` は無制限）。 |
| `--no-hooks` | `---->` | ロールバック中のフック実行を抑止します。 |
| `--recreate-pods` | `---->` | 該当する場合、リソースの Pod を再起動します。 |
| `--timeout=DURATION` | `---->` | 個々の Kubernetes 操作を待機する時間。 |
| `--wait` | `---->` | リリースを成功とする前にすべてのリソースが準備完了するまで待機します。 |
| `--wait-for-jobs` | `---->` | `--wait` と併用時、すべての Job が完了するまで待機します。 |

## Helm グローバルオプション

すべての [Helm グローバルオプション](status.md#helm-global-options)が Helm に転送されます。

## グローバルオプション

すべての[グローバルオプション](index.md)も利用できます。

## 例

```bash
# 直前のリビジョンへロールバック
kube-kts rollback my-app

# リビジョン 3 へロールバックし、準備完了を待機
kube-kts rollback my-app 3 --wait
```
